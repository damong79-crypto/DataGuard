# PRD — DB 정합성 자동 체크 데스크탑 앱 (가칭: DataGuard)

> 작성일: 2026-06-17
> 작성 도구: PRD 심층 인터뷰 (디테일 모드)

---

## 1. 프로젝트 개요

- **해결하려는 문제**: 관리 대상 DB·테이블이 늘면서 수작업 정합성 체크가 한계에 도달. 탐지 지연(늦은 발견 → 장애)과 추적·속인화 문제(이력 부재, 담당자 종속)가 모두 실제 사고로 이어진 경험이 있음.
- **배경**: 관리할 DB·테이블 증가로 사람이 일일이 챙기는 방식이 더 이상 감당 안 됨.
- **기대 변화**:
  - (현재) 사람이 기억해 수동 실행 → 늦은 발견·이력 없음·담당자 종속
  - (앞으로) 등록 쿼리가 스케줄 자동 실행 → 즉시 알림·이력 누적으로 추적·담당자 교체 시에도 그대로 인수인계

---

## 2. 사용자 분석

- **주 사용자**
  - **등록자(소수)**: SQL을 직접 다루는 DBA·데이터 엔지니어. 앱에서 쿼리·스케줄·수신자 관리.
  - **수신자(다수 가능)**: 결과를 이메일로만 수신(개발 리드·운영 담당·테이블 책임자). 앱 미사용.
- **사용 맥락**: 업무시간 중 등록자 PC에 트레이 상주, 내부 스케줄러로 실행. 셋업은 가끔, 평소엔 백그라운드 실행 + 가끔 이력 확인.
- **현재 해결 방식과 한계**: DBeaver·SSMS에 쿼리를 저장해두고 생각날 때 수동 실행. 쿼리 자산은 있으나 자동 실행·이력·알림이 전무.

---

## 3. 기능 요구사항

### 핵심 기능 (개발 우선순위 순)

**[MVP]**
1. **DB 연결 관리** — 체크 대상 DB(SQL Server·PostgreSQL) 접속정보 등록
2. **쿼리 등록** — 정합성 체크 쿼리 등록. 판정: 결과 0건=정상, 1건 이상=이상
3. **수동 실행("지금 실행")** — 등록 쿼리 즉시 실행·결과 확인
4. **이메일 알림** — 결과 발송. 쿼리별로 "이상일 때만 / 항상" 선택

**[그 위에 얹을 기능]**
5. **스케줄 설정** — 쿼리별 자동 실행 시점 (업무시간 중 상주 실행)
6. **결과 이력 저장** — 실행 결과 누적, 추적·인수인계 지원

### 차별점

기존 DB 클라이언트는 쿼리 저장 후 사람이 직접 실행 → 이 앱은 **자동 실행 + 0건 판정 + 이메일 알림 + 이력**을 한 번에 연결.

---

## 4. 비기능 요구사항

- **사용 환경**: Windows 데스크탑 앱(.NET WPF), 트레이 상주. 대상 DB: SQL Server + PostgreSQL.
- **데이터 민감도**:
  - DB 접속정보는 평문 저장 금지(Windows 자격 증명 관리자 또는 암호화 저장).
  - 쿼리 결과는 건수·요약만 발송(민감값 미포함).
- **제약**: 1인 개발, 예산 없음(사내용), 시간만 투입 가능 → MVP(1~4번) 우선, 스케줄·이력은 이후.

---

## 5. 성공 지표

- **성공 모습**: 수동 실행이 사라지고 등록 체크가 자동 실행되며, 문제를 장애 전에 조기 발견. 담당자 교체에도 인수인계 가능.
- **핵심 지표**:
  - [핵심] 조기 발견 건수(효과)
  - [보조] 자동 체크 중인 쿼리·테이블 수(활동량)

---

## 6. 사용자 여정 (User Journey)

### 등록자 — 최초 셋업
1. 앱 실행 → DB 연결 추가 (SQL Server/PostgreSQL 선택, 접속정보 입력) → 연결 테스트 → 자격증명은 Windows 자격 증명 관리자에 저장
2. 체크 쿼리 등록 → "틀어진 행을 뽑는" SQL 작성 → 판정규칙 확인(0건=정상) → "지금 실행"으로 동작 검증
3. 수신자 이메일 등록 + 발송 정책 선택(이상일 때만 / 항상)
4. 스케줄 설정 (예: 평일 09:00, 매시간) → 저장
5. 앱을 트레이로 최소화 → 상주 시작

### 평상시 — 자동 운영
6. 스케줄 시각 도래 → 앱이 쿼리 자동 실행 → 0건 판정
7. 이상(1건+) 발견 → 정책에 따라 수신자에게 이메일 발송
8. 실행 결과(성공/실패/건수/시각)가 이력에 누적

### 수신자
9. 이메일 수신 → "어떤 체크가 / 언제 / 몇 건 이상" 요약 확인 → 필요 시 등록자에게 후속 조치 요청

## 7. 기능별 상세 사양 (입력 → 동작 → 출력)

### ① DB 연결 관리
- **입력**: DB종류, 호스트, 포트, DB명, 계정, 비밀번호
- **동작**: 연결 테스트, 자격증명 암호화 저장(평문 금지)
- **출력**: 연결 목록(상태 표시), 성공/실패 메시지

### ② 쿼리 등록
- **입력**: 쿼리명, 대상 연결, SQL 본문, 발송 정책(이상시/항상)
- **동작**: 저장, 판정규칙 적용(결과 행 수 ≥ 1 → 이상)
- **출력**: 등록 쿼리 목록

### ③ 수동 실행("지금 실행")
- **입력**: 실행할 쿼리 선택
- **동작**: 해당 연결로 쿼리 실행, 결과 행 수 집계, 정상/이상 판정
- **출력**: 결과(정상/이상 + 건수 + 실행시각), 화면 즉시 표시

### ④ 이메일 알림
- **입력**: 수신자 주소, SMTP 설정(서버/포트/계정)
- **동작**: 정책·판정 결과에 따라 메일 생성·발송(건수·요약만 포함)
- **출력**: 발송 성공/실패 로그
- ※ SMTP 계정 정보도 암호화 저장 대상

### ⑤ 스케줄 설정
- **입력**: 쿼리별 실행 주기(시각/요일/간격)
- **동작**: 앱 내부 스케줄러가 트레이 상주 중 해당 시각에 트리거
- **출력**: 다음 실행 예정 시각 표시
- ※ 앱 미실행 시간대 체크는 건너뜀(상시 상주 전제)

### ⑥ 결과 이력 저장
- **입력**: (자동) 각 실행 결과
- **동작**: 로컬 DB(SQLite 등)에 결과 누적 저장
- **출력**: 쿼리별 실행 이력(시각/정상·이상/건수), 추세 확인

---

## 8. 단계별 개발 우선순위

### [MVP — 핵심 가치 1회 관통]
- DB 연결 1개 → 쿼리 1개 등록 → 수동 실행 → 이메일 발송
- 목표: "등록한 쿼리가 돌아서 메일이 온다"를 끝까지 검증

### [다음 단계]
- 스케줄 자동 실행(트레이 상주 스케줄러)
- 결과 이력 저장 + 조회 화면
- 발송 정책(이상시/항상) 쿼리별 분기

### [추후]
- 다중 연결·다중 쿼리 일괄 관리 UI 고도화
- 이력 기반 추세/대시보드(보조 지표 D 시각화)
- 체크 쿼리 가져오기/내보내기(인수인계 강화)

---

## 9. 위험 요소

### ① 자격증명 보안
DB·SMTP 비밀번호 저장이 최대 리스크. 평문/단순 인코딩 금지. Windows DPAPI(자격 증명 관리자) 활용 권장. 초기에 확정할 것.

### ② 스케줄 신뢰성 (상주형의 한계)
앱 종료·PC 절전 시 해당 체크 누락. 사용자가 "돌고 있다"고 착각할 위험 → 마지막 실행 시각 명확히 표시, 누락 감지 고려. (무인 시간대 요구가 커지면 OS 작업 스케줄러 위임으로 재설계)

### ③ 쿼리 오류·장기 실행
등록자가 짠 쿼리가 문법 오류·타임아웃·과부하를 유발 가능. 실행 타임아웃·에러 핸들링 필수. 잘못된 쿼리가 운영 DB에 부담 주지 않도록 **읽기 전용 계정** 권장.

### ④ 1인 개발 + 시간 제약
6개 기능 전부 욕심내면 미완성 위험. MVP 4개에 집중, 스케줄·이력은 동작 검증 후 단계적 추가.
